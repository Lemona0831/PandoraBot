# PANDORA 1.0 UX 점검 메모

이 문서는 “기능이 있냐 없냐”보다,  
**실제로 운영할 때 어디서 막히는지**를 기준으로 적어 둔 점검 메모입니다.

완성 선언 문서라기보다, 운영하면서 계속 손보는 쪽에 가깝습니다.  
한 번 쓰고 끝내는 문서가 아니라, 실제로 부딪힌 불편을 계속 적어두는 메모라고 보면 됩니다.

## 이번에 바로 손본 부분

### `/등록`을 비공개 응답으로 고정

원본 시트 URL이나 등록 과정이 채널에 그대로 보이는 건 운영상 부담이 큽니다.  
그래서 `/등록`은 비공개 응답으로 두는 쪽이 맞다고 보고 정리했습니다.

### `/등록`, `/갱신` 뒤에 다음 행동 안내 추가

등록은 했는데 그다음 뭘 해야 하는지 모르겠는 경우가 많았습니다.  
그래서 성공 메시지 뒤에 다음 흐름이 보이도록 정리했습니다.

- 등록 뒤: `/검수상태`
- 승인 뒤: `/선택`
- 선택 뒤: `/현재` 또는 `/정보`

### `/도움말`을 실제 흐름 기준으로 재정리

예전에는 기능 나열에 가까웠고, 처음 보는 사람 입장에서 조금 딱딱했습니다.  
지금은:

- 플레이어
- 판정
- 관리자

세 갈래로 나누고,  
“언제 쓰는지 / 나만 보는지 / 채널에 보여도 되는지”를 먼저 읽히게 정리했습니다.

### `/전투상태` 과다 출력 제한

전투 인원이 많아질수록 한 화면에 너무 많이 쏟아져서 읽기 힘들었습니다.  
그래서 표시 수를 제한하고, 더 있다는 안내를 붙였습니다.

### raw 에러 문구 완화

`Error: ...` 그대로 던지는 문장은 비개발자 운영자에게 친절하지 않습니다.  
가능한 범위에서:

- 지금 뭐가 문제인지
- 다음에 뭘 해보면 되는지

가 보이게 바꿨습니다.

## 지금 기준으로 괜찮아진 부분

### 등록 뒤 흐름

예전보다 훨씬 덜 끊깁니다.  
등록 -> 검수 -> 승인 -> 선택 -> 현재 확인 흐름이 이제 눈에 보입니다.

### 전투 세션 관련 안내

전투 세션이 없을 때, 이미 열려 있을 때, 종료 후 다시 시작할 때 메시지가 예전보다 덜 헷갈립니다.

### 운영자 문서화

운영자가 README만 보고 버티는 구조가 아니게 됐습니다.  
이제는 `docs/OPERATOR_GUIDE.md`를 먼저 보면 됩니다.

## 아직 더 다듬을 만한 부분

### 1. 중복 대상 안내

캐릭터 이름, 에너미 이름, 전투 참가자 이름이 겹칠 때  
“그래서 뭘 다시 넣어야 하는지”가 조금 더 친절하면 좋습니다.

특히 아래 명령은 후보 안내가 더 중요합니다.

- `/정보조회`
- `/에너미조회`
- `/전투피해`
- `/전투회복`
- `/전투퇴장`

### 2. `/전투시작` 뒤 추천 흐름

전투를 열고 나면 바로 뭘 해야 하는지 짧게 같이 안내해주면 더 좋습니다.

예를 들면:

- `/전투참가`
- `/에너미소환`
- `/전투상태`

이 정도를 붙여도 운영자는 꽤 편해집니다.

### 3. `/드롭테스트`의 성격 안내

이건 실전 보상 지급이 아니라 확률 체크용인데,  
처음 보는 사람은 `/드롭`이랑 헷갈릴 수 있습니다.

지금도 구분은 되어 있지만, 더 눈에 띄게 해도 괜찮습니다.

### 4. 전투 화면 요약/상세 분리

현재는 한 화면에 보여주는 편이라, 인원이 더 늘어나면 요약 화면과 상세 화면을 분리하는 쪽도 생각해볼 수 있습니다.

## 공개 / 비공개 구분 기준

### 나만 보는 쪽이 맞는 명령

- `/등록`
- `/갱신`
- `/검수상태`
- `/내정보`
- `/목록`
- `/선택`
- `/현재`
- `/해제`
- `/삭제`
- `/도움말`
- 대부분의 관리자 조회/전투 관리 명령

이유는 단순합니다.  
개인 캐릭터 정보나 운영 정리용 정보는 채널 전체에 흘러갈 필요가 없습니다.

### 채널에 보여도 되는 명령

- `/정보`
- `/판정`
- `/공지`
- `/피해`, `/회복`의 공개 알림
- `/드롭`

이쪽은 장면 공유나 운영 공지가 목적이라, 공개가 오히려 자연스럽습니다.

## DB와 Sheets 역할 정리

지금도 가장 헷갈리기 쉬운 부분이라, 계속 분명하게 적어둘 필요가 있습니다.

### Google Sheets가 하는 일

- 원본 캐릭터 시트 읽기
- `/등록`, `/갱신`
- 초기 이관 확인

### DB가 하는 일

- 캐릭터 조회/운영
- 검수/로그
- 에너미/드롭
- 전투 세션/참가자/상태

즉, 운영 중에는 DB를 본다고 생각하면 거의 맞습니다.

## 운영자 체크 포인트

운영자 기준으로는 아래 정도만 계속 보면 됩니다.

- 등록 뒤 다음 흐름이 자연스럽게 이어지는가
- 검수/승인/반려가 헷갈리지 않는가
- 전투 시작 -> 참가 -> 소환 -> 상태 확인 -> 종료가 한 번에 이어지는가
- 로그가 필요한 순간에 바로 보이는가
- 없는 데이터, 중복 데이터, 권한 문제 메시지가 너무 딱딱하지 않은가

## 개발 쪽에서 계속 볼 것

- raw 에러 문구 남아 있는지
- rate limit 안내가 충분한지
- DB 모드와 fallback 흔적이 사용자 경험을 해치지 않는지
- deprecated 경고 교체 계획
- 패키지 취약성 경고 대응 계획

## 한 줄 정리

PANDORA 1.0은 “기능이 많다”보다  
**운영자가 덜 막히고, 플레이어가 덜 헷갈리는가**가 더 중요합니다.

지금은 그 기준으로 꽤 많이 정리된 상태고,  
앞으로도 운영하면서 계속 손볼 문서라고 보면 됩니다.
