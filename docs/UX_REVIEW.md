# PANDORA 1.0 UX Review

## 목적
이 문서는 PANDORA 1.0 기준으로, 현재 구현된 기능을 실제 사용 흐름에서 다시 점검한 기록이다.
기능이 있는지만 보는 문서가 아니라, 사용자가 어디에서 막히는지, 진행자가 어디에서 헷갈릴 수 있는지, 개발자가 어디를 더 손봐야 하는지를 함께 정리한다.

## 이번 점검에서 바로 반영한 개선
- `/등록`을 비공개 응답으로 고정했다.
  - 원본 시트 URL과 등록 과정은 공개 채널보다 개인 확인이 더 안전하다.
- `/등록`, `/갱신` 성공 메시지에 다음 행동 안내를 추가했다.
  - 신규 등록 후: `/검수상태` 확인
  - 갱신 후: `/검수상태` 확인, 승인된 캐릭터라면 `/선택` -> `/현재` 또는 `/정보`
- `/도움말`을 플레이어 / 판정 / 관리자 기준으로 다시 구성했다.
  - 단순 기능 나열 대신, 언제 쓰는지 / 나만 보는지 / 채널에 보여도 되는지를 먼저 안내한다.
- `/전투상태`는 과다 출력 방지를 위해 표시 수를 제한하고, 초과 시 안내 문구를 추가했다.
  - 플레이어 최대 15명
  - 에너미 최대 20명
- 일부 raw 에러 메시지를 사용자 친화적으로 정리했다.
  - 캐릭터 미등록
  - 선택 캐릭터 없음
  - 승인 관련 안내
  - TooManyRequests / 429 계열 안내

## 현재 공개/비공개 구분 기준

### 나만 보는 것이 맞는 명령
- `/등록`
- `/갱신`
- `/검수상태`
- `/내정보`
- `/목록`
- `/선택`
- `/현재`
- `/해제`
- `/삭제`
- `/도움말`
- 대부분의 관리자 조회/전투 관리 명령

이유:
- 개인 캐릭터 정보, 승인 상태, 원본 시트 링크, 운영 정리 정보는 채널 전체보다 개인 응답이 더 안전하다.

### 채널에 보여도 되는 명령
- `/정보`
- `/판정`
- `/공지`
- `/피해`, `/회복`의 공개 알림
- `/드롭`

이유:
- 장면 공유, 판정 결과 공유, 상태 변화 안내, 보상 결과 공유는 채널 맥락 안에서 보는 편이 더 자연스럽다.

## 주요 마찰 지점 점검

### 1. 등록 후 다음 행동이 불명확함
상태:
- 1차 해결

문제:
- 등록이 끝나도 사용자는 다음에 검수를 기다려야 하는지, 바로 선택 가능한지 헷갈리기 쉽다.

현재 처리:
- 성공 메시지에 다음 행동 안내를 추가했다.

추가 권장:
- `/검수상태` 결과에도 "승인 후 `/선택` 가능" 문구를 더 강하게 넣으면 좋다.

### 2. 중복 대상 오류가 사용자를 멈추게 함
상태:
- 부분 해결

문제:
- 캐릭터명, 에너미명, 전투 참가자명이 겹칠 때 "누구를 다시 지정해야 하는지"가 모호해질 수 있다.

현재 처리:
- 전투 참가자/에너미 쪽은 후보 안내나 재확인 유도가 비교적 잘 들어가 있다.

추가 권장:
- 일반 캐릭터 관련 명령에도 후보 목록 또는 재지정 유도 방식이 더 넓게 통일되면 좋다.

### 3. 전투 세션 없음 오류
상태:
- 양호

문제:
- 전투 명령이 세션 기반이기 때문에, 세션이 없을 때 사용자는 무엇부터 해야 하는지 바로 알아야 한다.

현재 처리:
- `/전투시작` 안내가 나오도록 정리되어 있다.

추가 권장:
- `/전투시작` 성공 직후 `/전투참가`, `/에너미소환`, `/전투상태` 추천 흐름까지 함께 안내하면 더 좋다.

### 4. TooManyRequests 안내 부족
상태:
- 부분 해결

문제:
- 요청이 몰릴 때 사용자는 "왜 실패했는지"보다 "지금 멈추고 다시 눌러야 하는지"를 알고 싶다.

현재 처리:
- 상위 interaction 로그에 rate limit 힌트를 남기도록 보강했다.
- 일부 사용자 메시지에서도 429 / TooManyRequests 계열을 친화적으로 바꿨다.

추가 권장:
- 관리자 명령 전반에도 같은 문구 기준을 확장하면 좋다.

### 5. 전투 상태 과다 출력
상태:
- 1차 해결

문제:
- 인원이 많아질수록 `/전투상태`가 너무 길어져 읽기 어려워진다.

현재 처리:
- 표시 수 제한과 초과 안내 추가

추가 권장:
- 이후에는 요약/상세 분리나 페이지 처리도 고려할 수 있다.

### 6. DB / Sheets 역할 혼동
상태:
- 주의 필요

문제:
- 1.0 전환기라 저장소가 완전히 단일화되지 않았다.
- 운영자와 개발자가 "지금 무엇이 기준 데이터인지"를 혼동할 수 있다.

현재 기준:
- 캐릭터 등록, 검수, 선택, 기본 상태: Google Sheets 중심
- 에너미 / 드롭: repository를 통해 DB 우선, 미설정 시 Sheets fallback 가능
- 전투 세션, 전투 참가자, 전투 로그: DB 중심

권장:
- README와 운영 문서에 "전투는 DB, 캐릭터 원본 관리와 일부 운영은 아직 Sheets"라는 문장을 분명히 유지한다.

## 추가로 손볼 가치가 큰 지점
- `Error: {ex.Message}` 형태의 raw 예외 문구가 관리자 명령 쪽에 아직 많이 남아 있다.
- `/전투시작`, `/전투종료`, `/전투참가`, `/에너미소환` 성공 후 다음 행동 추천이 더 붙으면 흐름이 좋아진다.
- `/드롭테스트`는 테스트 전용 명령이라는 점이 더 선명하게 보여도 좋다.
- `/내정보`, `/검수상태`는 캐릭터 수가 많아질 때 가독성을 다시 점검할 필요가 있다.

## 운영자 점검 사항
- `/등록` 후 플레이어가 다음 행동을 이해하는지
- `/검수상태`만 보고도 승인 대기 / 승인 완료 / 반려를 충분히 구분하는지
- `/전투시작` 후 `/전투상태`가 자연스럽게 이어지는지
- `/전투참가`, `/에너미소환`에서 이름 중복 시 진행자가 재지정하기 쉬운지
- `/드롭` 결과가 active 전투 세션에서 `combat_logs`에 남는지
- 참가자가 많은 전투에서 `/전투상태`가 읽을 만한 길이인지

## 개발자 점검 사항
- 관리자 명령 전반의 raw 에러 문구 정리
- rate limit / interaction timeout 안내의 추가 확장
- DB 모드와 Sheets fallback 모드에서 UX 차이가 과도하지 않은지 확인
- deprecated 경고(`GoogleCredential.FromFile`) 교체 계획 수립
- package vulnerability warning(`System.Security.Cryptography.Xml`) 대응 계획 수립

## 결론
PANDORA 1.0은 기능 수보다 "지금 무엇을 하면 되는지 바로 알 수 있는가"가 더 중요하다.
현재 상태는 핵심 기능은 갖춰졌고, 전투 세션 흐름도 1차 연결되었다.
다만 개장 전까지는 아래 두 축을 계속 다듬는 편이 좋다.

- 다음 행동 안내 강화
- raw 에러 메시지의 사용자 친화성 강화

이 문서는 운영 테스트를 거치며 계속 업데이트하는 살아 있는 점검 문서로 유지한다.
